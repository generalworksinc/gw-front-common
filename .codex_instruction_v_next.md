# タスク: gw_front_common 非破壊改善リリース（A-1 authStoreクローン修正 / A-3 eventWithLoading JSDoc / B-2 createResettableStore）

対象リポジトリ: このリポジトリ（gw_front_common）のみ。テンプレートリポジトリは変更しない。
リポジトリの AGENTS.md / SOLID_COMPONENT.md の規約に従うこと。**全て非破壊（既存APIの挙動・シグネチャを変えない）**。

## 背景

generalworks の solid/nuxt テンプレート開発で実際に踏んだバグの原本・温床がライブラリ側にある:
1. `createStore(defaultState)` はクローンしないため defaultState が汚染され `reset()` が no-op になる（ログアウト無効バグの真因）
2. `eventWithLoading` はクリック連打ガード専用なのに、初期データ取得に誤用されて「isLoading中は黙ってスキップ」により初回fetchが消えるバグを生んだ
3. Solid 2.0（v2.0.0-beta.0）で「書き込みはmicrotaskバッチ化・読み取りはflushまで旧値」になるため、set直後に同期でstoreを読み戻して永続化する実装（makePersisted型）は将来壊れる

## 変更1 (A-1): solid/features/auth/authStore.ts のクローン罠修正

```ts
// 現状（バグ）:
const [store, setStore] = createStore<AuthUser>(defaultState);

// 修正:
// createStore は渡したオブジェクトをクローンせず内部状態として使う。
// defaultState を直接渡すと書き込みで defaultState 自体が汚染され、
// reset() の { ...defaultState } が「現在値の書き戻し = no-op」になる。必ずコピーを渡す。
const [store, setStore] = createStore<AuthUser>({ ...defaultState });
```

コメントも上記の趣旨で残す。

## 変更2 (A-3): eventWithLoading の JSDoc と内部整理（vue / solid 両方）

対象: `vue/features/loading/utils.ts` と `solid/features/loading/utils.ts`（同内容で）。

1. JSDoc を追加（要旨、日本語）:
   - **用途はユーザー操作（クリック等）の連打ガード専用**。グローバルローディング表示+実行中は後続操作を無効化する
   - **isLoading 中に呼ばれた場合は何も実行せず `false` を返す**（呼び出し側はスキップされ得ることを前提にする）
   - **初期ロード・リアクティブなデータ取得に使ってはならない**。無関係なロードと排他されて取得が黙って消える。データ取得はコンポーネント/ストアのローカルな読み込み状態で管理する
   - `setTimeout` はローディング表示の描画を先に確定させるための意図的なもの（コメント化）
   - Solid 2.0 の変更（microtaskバッチ）による影響はこの関数にはない
2. 内部整理: `resolve(Promise.reject(err))` は紛らわしいので `reject(err)` に変更（await する呼び出し側から見た挙動は同一 = 非破壊）。then/catch のネストが素直になるなら整理してよいが、**「busy時は false を返す」「結果を返す」「エラーは throw される」という外部契約は一切変えない**。
3. `awaitLoadingWith` にも1行 JSDoc（eventWithLoading をイベントハンドラ化するヘルパーである旨）。

## 変更3 (B-2): solid に createResettableStore を新規追加

配置: `solid/features/store/resettableStore.ts`（features 配下の慣例に合わせる。既存の export 集約ファイル（solid の mod / index）から `createResettableStore` を named export すること）。

### 設計（このとおりに実装）

```ts
import { createStore, reconcile, unwrap, type SetStoreFunction } from 'solid-js/store';
import { isServer } from 'solid-js/web';

export type ResettableStoreOptions = {
	/** 指定するとクライアントで localStorage（または storage）へ永続化する */
	persist?: {
		name: string;
		storage?: Pick<Storage, 'getItem' | 'setItem' | 'removeItem'>;
	};
};

/**
 * リセット可能（+任意で永続化）なストアを作るファクトリ。
 *
 * createStore の2つの罠を構造的に回避する:
 * 1. createStore は初期値をクローンしない → 内部でディープコピーして保持するため、
 *    呼び出し側の defaultState がストア書き込みで汚染されることがない
 * 2. reset() は保持しておいた初期スナップショットを reconcile で書き戻すため、
 *    「汚染された初期値の書き戻し no-op」が起きない
 *
 * 永続化は makePersisted と異なり「set 直後の同期読み戻し」に依存しない
 * （microtask で遅延して書き込む）。Solid 2.0 の microtask バッチ化
 * （読み取りは flush まで旧値）でも正しい値が永続化される。
 */
export function createResettableStore<T extends object>(
	defaultState: T,
	options?: ResettableStoreOptions,
): { store: T; set: SetStoreFunction<T>; reset: () => void } {
	const snapshot = structuredClone(defaultState);
	const [store, setStore] = createStore<T>(structuredClone(snapshot));

	const persist = options?.persist;
	const storage = persist ? (persist.storage ?? (isServer ? undefined : globalThis.localStorage)) : undefined;

	// 復元（クライアントのみ・作成時に1回）
	if (persist && storage) {
		try {
			const raw = storage.getItem(persist.name);
			if (raw != null) setStore(reconcile({ ...snapshot, ...JSON.parse(raw) }));
		} catch { /* 壊れた保存値は無視して初期状態で開始 */ }
	}

	// 書き込みは microtask で遅延+同一tick内は1回に集約（Solid 2.0 のバッチ化でも安全）
	let scheduled = false;
	const schedulePersist = () => {
		if (!persist || !storage || scheduled) return;
		scheduled = true;
		queueMicrotask(() => {
			scheduled = false;
			try {
				storage.setItem(persist.name, JSON.stringify(unwrap(store)));
			} catch { /* ストレージ不可（容量等）は黙殺しない方が良いが、UIを壊さないため warn に留める */ }
		});
	};

	const set: SetStoreFunction<T> = ((...args: any[]) => {
		(setStore as any)(...args);
		schedulePersist();
	}) as SetStoreFunction<T>;

	const reset = () => {
		setStore(reconcile(structuredClone(snapshot)));
		schedulePersist();
	};

	return { store, set, reset };
}
```

注意点:
- makePersisted は使わない（同期読み戻し依存を避けるため）。@solid-primitives/storage への依存はこのファイルには持ち込まない。
- catch 節の方針: 復元失敗は無視、書き込み失敗は `console.warn`（isDev 判定があればそれに合わせる）。
- 型は any キャストを最小に。biome の lint が通る形に調整してよいが、公開シグネチャは上記のとおり。

### authStore のリファクタ（同ファイル内で利用）

`solid/features/auth/authStore.ts` を createResettableStore ベースに書き換えてよい（**公開API `authStore = { get, set, reset, isLoggedIn }` と挙動は不変**であること）。makePersisted への依存が消える。localStorage キー名 `authStore` は維持（既存アプリの保存データと互換）。
※ 既存の保存形式は JSON そのままなので互換になるはず。復元時のマージ（`{...snapshot, ...parsed}`）でキー欠けにも耐える。

## テスト（__tests__ に追加。bun test 形式、既存テストの流儀に合わせる）

1. `resettable_store.test.ts`（新規）:
   - defaultState が汚染されないこと（set 後に defaultState の中身が変わっていない）
   - set → reset で初期状態に戻ること（クローン罠の回帰テスト: **defaultState を直接渡す旧パターンで壊れていたシナリオ**を再現する形で）
   - persist 指定時: モック storage で、set 後 microtask 経過（`await Promise.resolve()` 等）で書き込まれること／同一tick内の複数 set で書き込みが1回に集約されること／作成時に保存値から復元されること／壊れた JSON でもthrowしないこと
   - persist なしなら storage に一切触れないこと
2. authStore の回帰テスト（既存テストがあれば拡張、なければ新規）: set → reset → isLoggedIn() === false、get().fullName が null に戻る。
3. eventWithLoading の既存テスト（solid_loading_utils.test.js）が通ることを確認。エラー経路（async 関数が reject）で throw されることのテストがなければ1ケース追加。
